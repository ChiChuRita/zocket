1. Overview

Zocket lets developers describe a realtime application as a collection of actors: small, stateful server-side objects with methods, events, and typed state schemas. Clients connect over WebSockets and interact with actors through RPC-style calls while receiving live state updates as JSON Patches.

Design goals

  • End-to-end type safety — actor schemas authored once with Zod; client types are inferred directly from the server definition.
  • Sequential execution per actor — no interleaving, no accidental races inside a single actor instance.
  • State sync via JSON Patch — mutations are produced through Immer and shipped as minimal patches to subscribers.
  • Pub/sub events — typed, broadcast-only channels for one-to-many signaling.
  • Deploy anywhere — run self-hosted with Docker Compose, or push bundles to the hosted platform.

RPC correlation

Client sends { id, type: "call", actor, actorId, method, input } on inbound.>. The runtime invokes the actor method and replies with { id, type: "result", ... } on outbound.{...sessionId}. The gateway matches by sessionId (captured in the inbound envelope) and forwards to the right WebSocket; the client correlates by id.

Inside the runtime

The runtime does not spin up a NATS subscription per actor instance. It runs one durable push consumer per actor type, interleaves messages for every {actorId}, and funnels each one into a per-instance FIFO queue. Client-initiated state and event subscriptions are pure in-memory Set<Connection> references on the actor — broadcasts are fanned out by the runtime, not by NATS subject matching.

What happens when a client calls a method

  1. Gateway publishes InboundEnvelope to inbound.{ws}.{proj}.{actorType}.{actorId}.
  2. The runtime's durable consumer for that {actorType} receives it (one consumer serves all actorIds of that type — messages are interleaved).
  3. The dispatcher resolves the actor instance by {actorId}, creating it lazily (onActivate on first hit), and pushes the envelope onto that instance's FIFO queue.
  4. If the instance isn't already processing, it drains its queue one message at a time. There is no mutex — sequentiality is an invariant of the single-drain loop. Different actor instances run concurrently; the same instance never does.
  5. Method body runs, producing state mutations through an Immer draft. The runtime diffs the draft into a JSON Patch.
  6. The JetStream message is acked.

How subscriptions and fan-out work

  • Subscriptions are not NATS subjects. A client subscribe-state or subscribe-event message mutates an in-memory Set<Connection> on the actor instance. No new consumer, no new subject — just a pointer added to a set.
  • State fan-out. After each state mutation, the runtime iterates stateSubscribers and calls conn.send(patch) on each. Each VirtualConnection publishes one JetStream message to outbound.{ws}.{proj}.{sessionId}. Three subscribers means three outbound messages — fan-out is done in the runtime, not by NATS subject matching.
  • Event fan-out. Identical mechanism, over the eventSubscribers set.
  • Initial snapshot. On subscribe, the runtime immediately sends the current state to that one connection before the next mutation fires.

Actor and session lifecycle

  • Actor creation: lazy on first inbound message for an {actorId}. onActivate() fires once.
  • Actor eviction: none during normal operation. Instances stay hot until the runtime stops. State is not persisted — a restart drops all in-memory state and rebuilds it as messages arrive again (JetStream redelivers unacked work).
  • Session disconnect: the gateway publishes session.disconnected on core NATS; the runtime's session subscription callback removes that VirtualConnection from every actor's subscriber sets and fires onDisconnect(conn) hooks. The actor itself is untouched.

Entry points: packages/runtime/src/consumer.ts (consumer setup, session cleanup) · packages/server/src/runtime.ts (dispatcher, actor instance, subscriber sets, fan-out) · packages/runtime/src/virtual-connection.ts (outbound publish).

3. Components

Three runtime roles in the data path, plus the control plane off to the side.

Client

Browser-side SDK (@zocket/client, @zocket/react). Opens one WebSocket, sends typed calls, subscribes to state and events, applies JSON Patches.

Gateway

WebSocket edge (@zocket/gateway, :3001). Authorizes with the control plane, translates WS ↔ NATS, runs one ephemeral outbound consumer per session.

Runtime

Actor host (@zocket/runtime). One durable consumer per actor type, executes methods, owns Immer state, publishes patches and events.

Control plane

SaaS platform (platform/). Auth, projects, deployments, bundle storage. Talks REST, not NATS.

4. Hosted platform

The hosted platform under platform/ is a full-stack SaaS control plane built with React 19 and TanStack Router, deployed to Cloudflare Workers and backed by Neon Postgres.

Responsibilities

  • User and workspace management via WorkOS AuthKit
  • Project CRUD and deployment-token lifecycle
  • Actor bundle storage in S3
  • Runtime lifecycle orchestration (via Pulumi stacks)
  • API endpoints for CLI authentication (device flow), deployment uploads, and project queries

Database schema

Drizzle ORM, Postgres on Neon. Core tables: users, workspaces, workspace_memberships, projects, deployments (with actor_types, status, bundleKey), deploy_tokens, project_auth_configs (JWT/JWKS), runtime_configs, device_flows, cli_tokens.

Entry points: platform/src/server/api/app.ts · platform/src/server/db/schema.ts

5. Request flow

  1. Connect: client opens a WebSocket to the gateway on :3001 with an auth token.
  2. Authorize: gateway calls the control plane's /registry/authorize endpoint to validate the token and project.
  3. Publish: the gateway serializes the call and writes it to the NATS inbound stream.
  4. Execute: the runtime consumes the message, routes it to the correct actor instance, and runs the method body.
  5. Mutate: state changes run through Immer, producing a minimal JSON Patch.
  6. Broadcast: the patch is published to the outbound stream; subscribed gateways pick it up.
  7. Deliver: the gateway fan-outs the patch (plus any RPC reply) to every subscribed client.

6. Deployment flow

  1. CLI builds a bundle and uploads it to the platform API.
  2. Platform stores the artifact in S3 and writes a deployments row in Neon.
  3. Pulumi's project-runtime stack is invoked to provision or update the runtime.
  4. Runtime fetches the bundle from S3 and loads the actor types.
  5. Runtime reports status (ready / error) back to the platform API, which updates the deployment row.

7. Infrastructure

Pulumi stacks

pulumi-aws

Hosts runtimes in AWS (EC2 / container orchestration).

pulumi-cloudflare

DNS, edge routing, and Cloudflare Workers for the platform.

pulumi-neon

Provisions the serverless Postgres database for the platform.

pulumi-project-runtime

Per-project runtime stack; created on deploy to isolate tenant workloads.

Local development

Running docker compose up boots NATS (with JetStream), the gateway, and a runtime — all sharing the same NATS instance. See docker-compose.yml.
