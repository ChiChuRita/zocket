Zocket — Scaling Deployment Overview

Gateways · NATS · Firecracker runners · JetStream reliability

1. Bun Gateway Fleet (WebSocket termination)

The @zocket/gateway runs as a stateless Bun fleet behind a load balancer. Each gateway only terminates client WebSockets — it holds no actor state, so clients can land on any node. On connect it authenticates the session, attaches a {workspaceId, projectId, sessionId, userId, claims} scope to every frame, and translates between the wire protocol and NATS envelopes. Because gateways are stateless, the fleet scales horizontally by CPU / connection count, and one gateway dying only drops its live sockets — the client reconnects to another node and resumes.

2. NATS Cluster (routing + replication)

A clustered NATS deployment sits between gateways and runners. Routing is done entirely by subject hierarchy, not by point-to-point addressing:

  • Inbound: inbound.{workspaceId}.{projectId}.{actorType}.{actorId}
  • Outbound: outbound.{workspaceId}.{projectId}.{sessionId}
  • Session lifecycle (core NATS): session.connected.* / session.disconnected.*

NATS guarantees that whichever runtime is subscribed to that subject pattern receives the message, regardless of which gateway produced it. The two core subject trees are backed by JetStream streams replicated (R=3) across cluster nodes via Raft — a node loss doesn't lose in-flight messages. Per-actor ordering is preserved because a single durable consumer drains each {workspace, project, actorType} partition sequentially.

3. Actor Runners (Firecracker-isolated per tenant)

Actor code is untrusted tenant code, so each tenant gets dedicated @zocket/runtime processes running inside Firecracker microVMs on a runner host. The orchestrator places microVMs by tenant (DEPLOYMENT_ID), giving hardware-level isolation between workspaces while keeping cold-start in the hundreds-of-ms range. Runners scale independently of gateways — hot tenants get more VMs, idle ones get scaled to zero.

Today's v1 ships with dedicated runtime processes as the isolation boundary; Firecracker is the v2 hardening layer swapping the process boundary for a VM boundary without changing the NATS-facing contract.

4. Actor Runtime (instances + NATS subscription)

Inside each microVM, the runtime's ConsumerManager creates one durable JetStream consumer per actor type: rt-{workspaceId}-{projectId}-{actorType}. Incoming envelopes are dispatched to the matching actor instance (spawned lazily, keyed by actorId), which runs handlers sequentially against Immer-managed state. When an actor replies, it calls VirtualConnection.send(), which publishes to the outbound.* stream — the actor never knows which gateway owns the socket. Gateways attach ephemeral outbound consumers scoped to their live sessions; NATS fans the message to exactly the right gateway.

5. JetStream Reliability

  • INBOUND — workqueue retention, 60s max-age, explicit ACK. A message is only removed once the actor has processed it. If the runtime crashes mid-handler, JetStream redelivers to the replacement VM and the actor resumes from its last acked op. No client message is silently dropped.
  • OUTBOUND — limits retention, 30s max-age, ephemeral per-session consumers. When a client's WebSocket drops, the consumer goes away but outbound frames keep accumulating in the stream. On reconnect, the new gateway recreates an ephemeral consumer from the last-delivered sequence and drains the backlog — brief network blips don't lose server-to-client messages.
